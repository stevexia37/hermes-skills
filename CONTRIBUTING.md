## Contributing

Contributions should improve reuse, clarity, and public safety.

## What To Contribute

- new reusable Hermes skills
- skill hardening and cleanup
- stronger examples and documentation
- environment-variable guidance
- safety notes and edge-case handling

## Do Not Submit

- private credentials
- personal names or private routing metadata
- machine-specific absolute paths
- unreviewed financial automation that looks like guaranteed alpha
- organization-specific delivery rules without generalization

## Skill Review Checklist

- is the skill reusable outside one local setup
- does it avoid private IDs and secrets
- does it explain required environment variables
- does it contain at least one practical workflow or integration pattern
- does it describe key pitfalls
- if risky, does it include an explicit safety note

## Preferred Style

- concise frontmatter
- practical steps over vague theory
- concrete examples
- operational caveats where needed
- clear trigger conditions

## Pull Requests

PRs should include:

- what changed
- why it improves public reuse
- any safety or compatibility implications
